## 引言

分析循环是DeepTutor系统中“双循环”架构（Dual-Loop Architecture）的首要阶段，其核心目标是为后续的解题过程（Solve Loop）准备充足且精准的知识“弹药”。它通过一个由`InvestigateAgent`（调查员）和`NoteAgent`（笔记员）驱动的迭代过程，主动探索知识库，填补用户问题与现有知识之间的缺口。本文档将深入剖析其内部机制，从实现细节到配置参数，为开发者和使用者提供全面的技术指南。

## 核心组件与领域模型

分析循环的核心由两个智能体（Agent）和两个内存（Memory）构成，它们协同工作，形成一个闭环。

### InvestigateAgent（调查员）

`InvestigateAgent`是分析循环的“大脑”，负责决策“查询什么”和“如何查询”。它根据当前的知识状态和用户问题，生成具体的查询计划。

**关键实现细节**：
- **初始化**：在`__init__`方法中，它从全局配置中读取`tools.web_search.enabled`来决定是否启用网络搜索，并从`solve.agents.investigate_agent`配置节中获取`max_actions_per_round`（每轮最大动作数）和`max_iterations`（最大迭代次数）。
- **处理流程**：`process`方法是其核心。它首先构建包含用户问题、已有知识链和剩余问题的上下文，然后调用大语言模型（LLM）生成一个JSON格式的计划。该计划包含一个或多个工具调用（`rag_naive`, `rag_hybrid`, `web_search`, `query_item`）或一个终止信号（`none`）。
- **工具调用**：`_execute_single_action`方法负责执行具体的工具调用。它会记录工具调用的耗时，并将原始结果注册到`CitationMemory`中，生成一个唯一的引用ID（`cite_id`），然后创建一个`KnowledgeItem`对象。

**接口与参数**：
- `process`方法的输入参数包括`question`（用户问题）、`memory`（`InvestigateMemory`实例）、`citation_memory`（`CitationMemory`实例）等。
- 返回值是一个字典，包含`reasoning`（推理过程）、`should_stop`（是否应停止）、`knowledge_item_ids`（新生成的知识项ID列表）和`actions`（执行的动作列表）。

### NoteAgent（笔记员）

`NoteAgent`是分析循环的“速记员”，负责将`InvestigateAgent`获取的原始信息转化为结构化的、易于理解的知识笔记。

**关键实现细节**：
- **处理流程**：`process`方法接收一个或多个新知识项的ID。它会为每个ID获取对应的`KnowledgeItem`，然后调用LLM，根据`note_agent.yaml`中的提示词，将`raw_result`（原始结果）提炼成`summary`（摘要）。
- **摘要规则**：提示词中明确规定，`summary`必须是纯文本，禁止包含LaTeX公式，以确保其可读性。
- **引用更新**：生成摘要后，`NoteAgent`会调用`citation_memory.update_citation`方法，用新的`summary`内容更新`CitationMemory`中对应的条目。

**接口与参数**：
- `process`方法的输入参数包括`new_knowledge_ids`（需要处理的新知识项ID列表）。
- 返回值是一个字典，包含处理成功或失败的详细信息。

### InvestigateMemory（分析内存）

`InvestigateMemory`是分析循环的“知识仓库”，用于持久化存储迭代过程中产生的所有知识和元数据。

**领域模型**：
- `KnowledgeItem`：这是核心数据结构，代表一个知识单元。它包含`cite_id`（引用ID）、`tool_type`（工具类型）、`query`（查询内容）、`raw_result`（原始结果）和`summary`（摘要）等字段。
- `Reflections`：用于存储在分析过程中产生的“剩余问题”，即系统认为还需要进一步探究的问题。
- `knowledge_chain`：一个`KnowledgeItem`对象的列表，按时间顺序记录了所有获取的知识。
- `metadata`：一个字典，用于存储统计信息，如迭代次数、覆盖率等。

**功能**：
- 提供`add_knowledge`和`update_knowledge_summary`方法供`InvestigateAgent`和`NoteAgent`调用。
- 支持从文件加载（`load_or_create`）和保存（`save`）状态，确保过程可中断和恢复。

**Section sources**
- [investigate_memory.py](file://src/agents/solve/memory/investigate_memory.py#L13-L227)

### CitationMemory（引用内存）

`CitationMemory`是系统的“全局引用管理器”，它为所有工具调用（包括分析循环和解题循环）提供统一的引用ID生成和管理。

**领域模型**：
- `CitationItem`：代表一个引用条目，包含`cite_id`（如`[rag-1]`）、`tool_type`、`query`、`raw_result`、`source`（来源）和`content`（内容摘要）等。
- `tool_counters`：一个字典，用于跟踪每种工具类型（如`rag`, `web`）已生成的ID数量，以确保ID的唯一性和连续性。

**功能**：
- `add_citation`方法：根据`tool_type`生成唯一的`cite_id`（例如，`rag_naive`和`rag_hybrid`都使用`rag`前缀），并创建新的引用条目。
- `update_citation`方法：允许其他组件（如`NoteAgent`）更新引用条目的内容（如`content`字段）。
- `format_citations_markdown`方法：将引用列表格式化为Markdown，用于最终答案的输出。

## 分析循环的调用关系与工作流程

分析循环的执行流程是一个清晰的“调查-笔记”迭代过程，由`MainSolver`控制器驱动。

### 整体工作流程

```mermaid
sequenceDiagram
participant MainSolver as MainSolver
participant InvestigateAgent as InvestigateAgent
participant NoteAgent as NoteAgent
participant Tools as Tools (RAG, Web, etc.)
participant InvestigateMemory as InvestigateMemory
participant CitationMemory as CitationMemory
MainSolver->>InvestigateMemory : load_or_create(初始化)
MainSolver->>CitationMemory : load_or_create(初始化)
loop 每一轮迭代
MainSolver->>InvestigateAgent : process(问题, 内存, 引用内存)
InvestigateAgent->>InvestigateAgent : _build_context(构建上下文)
InvestigateAgent->>InvestigateAgent : call_llm(生成查询计划)
InvestigateAgent->>InvestigateAgent : 解析JSON计划
alt 计划非空且不为"none"
InvestigateAgent->>Tools : 执行工具调用(rag_search等)
Tools-->>InvestigateAgent : 返回原始结果
InvestigateAgent->>CitationMemory : add_citation(注册引用)
InvestigateAgent->>InvestigateMemory : add_knowledge(添加知识)
else 计划为空或为"none"
InvestigateAgent-->>MainSolver : should_stop=True
break 终止循环
end
alt 有新知识
MainSolver->>NoteAgent : process(新知识ID列表)
NoteAgent->>NoteAgent : 获取KnowledgeItem
NoteAgent->>NoteAgent : call_llm(生成摘要)
NoteAgent->>CitationMemory : update_citation(更新摘要)
NoteAgent->>InvestigateMemory : update_knowledge_summary(更新内存)
end
MainSolver->>InvestigateMemory : 更新元数据并保存
end
MainSolver->>MainSolver : 进入解题循环
```

### InvestigateAgent如何生成查询并调用工具

1.  **上下文构建**：`_build_context`方法将用户问题、完整的知识链（`knowledge_chain_full`）和剩余问题（`reflections_summary`）整合成一个详细的上下文。
2.  **提示词构建**：`_build_system_prompt`和`_build_user_prompt`方法根据配置动态构建系统提示和用户提示。系统提示定义了`InvestigateAgent`的角色、原则和工具使用指南。如果`web_search`被禁用，系统会自动从提示词中移除相关描述。
3.  **LLM调用与计划生成**：`process`方法调用LLM，期望返回一个JSON对象。LLM根据提示词和上下文，决定下一步的行动。
4.  **工具执行**：`_execute_single_action`方法根据计划中的`tool_type`，调用相应的工具函数：
    - `rag_naive`/`rag_hybrid`: 调用`rag_search`函数。
    - `web_search`: 调用`web_search`函数。
    - `query_item`: 调用`query_numbered_item`函数。
5.  **结果处理**：工具返回结果后，`InvestigateAgent`将其注册到`CitationMemory`，生成`cite_id`，并创建`KnowledgeItem`存入`InvestigateMemory`。

**Section sources**
- [investigate_agent.py](file://src/agents/solve/analysis_loop/investigate_agent.py#L44-L187)

### NoteAgent如何处理新知识并生成笔记

1.  **批量处理**：`process`方法可以接收一个新知识项ID列表，对每个ID进行循环处理。
2.  **上下文构建**：`_build_context`方法为每个`KnowledgeItem`构建一个包含其`raw_result`的上下文。
3.  **摘要生成**：`process`方法调用LLM，根据`note_agent.yaml`中的提示词，将`raw_result`提炼成`summary`。提示词强制要求`summary`为纯文本。
4.  **结果验证**：代码中使用`validate_note_output`函数和`ParseError`来确保LLM返回的是有效的JSON。
5.  **信息更新**：生成摘要后，`NoteAgent`会同时更新`CitationMemory`和`InvestigateMemory`中的对应条目，确保数据一致性。

## 配置选项与参数详解

分析循环的行为高度依赖于配置文件，主要涉及`config/main.yaml`和`config/agents.yaml`。

### 主要配置项

| 配置项 | 文件路径 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- |
| `max_iterations` | `solve.agents.investigate_agent.max_iterations` | `3` | 分析循环的最大迭代次数。超过此次数，即使知识不完整，循环也会强制进入解题阶段。 |
| `max_actions_per_round` | `solve.agents.investigate_agent.max_actions_per_round` | `1` | 每轮迭代中，`InvestigateAgent`最多可以执行的工具调用次数。 |
| `web_search.enabled` | `tools.web_search.enabled` | `true` | 全局开关，控制`InvestigateAgent`是否可以使用`web_search`工具。 |
| `temperature` | `solve.temperature` | `0.3` | 控制`InvestigateAgent`和`NoteAgent`的LLM输出随机性。较低的值（如0.3）使输出更确定和聚焦。 |
| `max_tokens` | `solve.max_tokens` | `8192` | LLM调用时允许的最大输出token数。 |

## 与其他组件的关系

分析循环并非孤立运行，它与系统的多个核心组件紧密协作。

### 与知识库（Knowledge Base）的关系

分析循环通过多种工具与知识库交互：
- **RAG工具**：`rag_search`函数是与知识库交互的主要方式，它利用向量检索和图检索技术，从`./data/knowledge_bases`目录下的知识库中查找相关信息。
- **编号项查询工具**：`query_numbered_item`函数允许直接通过ID（如"Definition 1.1"）精确查询知识库中的特定条目，这对于获取定义、定理和公式非常高效。

**Section sources**
- [rag_tool.py](file://src/tools/rag_tool.py#L31-L263)
- [query_item_tool.py](file://src/tools/query_item_tool.py#L15-L310)

### 与引用内存（CitationMemory）的关系

这是分析循环最核心的依赖之一。
- **引用ID生成**：`InvestigateAgent`在每次工具调用后，必须调用`CitationMemory.add_citation`来获取一个全局唯一的`cite_id`。这个ID会作为`KnowledgeItem`的一部分被存储。
- **信息更新**：`NoteAgent`在生成摘要后，会调用`CitationMemory.update_citation`来更新`cite_id`对应条目的`content`字段。
- **最终输出**：在解题循环的最后，系统会调用`CitationMemory.format_citations_markdown`，根据最终答案中使用的`cite_id`列表，生成完整的参考文献部分。

## 常见问题与解决方案

### 问题1：分析循环未收敛（迭代次数达到上限但知识仍不完整）

**现象**：日志显示`max_iterations(3) reached`，但解题过程可能因信息不足而失败。

**原因分析**：
1.  **知识库覆盖不足**：用户问题涉及的知识点在当前知识库中不存在。
2.  **查询策略失效**：`InvestigateAgent`生成的查询不够精准，未能命中关键信息。
3.  **配置限制**：`max_iterations`设置过低。

**解决方案**：
1.  **检查知识库**：确认知识库（如`ai_textbook`）是否包含相关领域的知识。必要时，使用`knowledge`模块添加新文档。
2.  **调整提示词**：检查`investigate_agent.yaml`中的系统提示，确保其指导原则（如“精准狙击”）能有效引导LLM生成高质量查询。
3.  **增加迭代次数**：在`config/main.yaml`中适当提高`solve.agents.investigate_agent.max_iterations`的值。
4.  **启用网络搜索**：确保`tools.web_search.enabled`为`true`，允许系统从外部网络获取最新信息。

### 问题2：知识覆盖不足或查询结果质量低

**现象**：`InvestigateAgent`返回了结果，但内容与问题无关或信息量不足。

**原因分析**：
1.  **RAG检索效果差**：知识库的索引质量不高，或RAG的检索模式（`naive`/`hybrid`）不适合当前查询。
2.  **查询语句模糊**：`InvestigateAgent`生成的`query`过于宽泛。

**解决方案**：
1.  **优化知识库**：确保知识库文档已正确分块和索引。可以尝试重新初始化知识库。
2.  **调整RAG模式**：在`investigate_agent.yaml`中，根据查询类型选择更合适的工具（`rag_naive`用于简单查询，`rag_hybrid`用于复杂关联查询）。
3.  **审查提示词**：强化提示词中关于“构建原子化、精准查询”的指导。

## 总结

分析循环是DeepTutor系统智能性的关键体现。它通过`InvestigateAgent`和`NoteAgent`的协同，实现了从被动响应到主动探索的转变。其设计精巧，通过`InvestigateMemory`和`CitationMemory`实现了状态的持久化和引用的统一管理。通过深入理解其工作流程、配置参数和与其他组件的交互，开发者可以有效地调试和优化系统性能，确保为最终的解题过程提供坚实的知识基础。
